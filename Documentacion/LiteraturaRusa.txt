\section{Introduction}

La aplicación "Literatura Rusa" fue creada en Flutter y Springboot. El tema "Literatura Rusa" fue elegido por el interés en obras de dostoyeski con la finalidad de irse extendiendo poco a poco en este genero literario. El objetivo fue pasar el proyecto de tweeter, adaptarlo a la aplicación, hacer funcional las reacciones y comentarios. Ademas permitir que se guarde a la base de datos, guardar el conteo de las reacciones y los usuarios que hicieron la publicacion. Como nube ocupamos claudinary y base de datos Aiven en PostgreSQL

\subsection{Patrones estructurales}

\subsubsection{FACADE}

Este se ocupo principalmente en flutter en la carpeta facade en el archivo library-facade.dart, este se utilizó para simplificar la comunicación entre las pantallas y los servicios. Este en lugar de que HomeScreen o LoginScreen llamen directamente a AuthService o BookService, llaman a métodos de LibraryFacade. Este tiene una ventaja es que el sistema ya no necesita conocer todos los detalles internos de los servicios solo lo llama y ya.

\subsection{Patrones creacionales}

Sirve para crear los objetos principales que necesita la aplicación, por ejemplo: AuthService, BookService y LibraryFacade.La idea es que en vez de crear esos servicios directamente en main.dart, se hace todo en una sola clase, es decir, main.dart no se preocupa por crear primero AuthService, luego BookService, y después pasarlos a LibraryFacade. Todo eso lo hace la fábrica.

\subsection{Singleton}

Se utilizó para mantener una única instancia de los servicios principales. Esto es importante especialmente en AuthService, porque ahí se guarda información de sesión como los tokens, la autentificacion de usuario

\subsection{Patrones de acceso a datos}
\subsubsection{Repository}
Se utilizó para separar el acceso a la base de datos de los controladores. Aqui BookPostRepository permite guardar y consultar libros, BookCommentRepository permite obtener comentarios asociados a un libro y BookReactionRepository permite guardar reacciones y contar cuántas tiene un libro.

\subsubsection{Instancia única}
El patrón Singleton se utilizó en Flutter en servicios como: AuthService, BookService. Este patrón permite que exista una sola instancia compartida de una clase.Este se usó porque AuthService maneja información importante de sesión, como el token JWT y el usuario autenticado. Si se crearan varias instancias diferentes de AuthService, podría haber inconsistencias, por ejemplo, que una instancia tenga token y otra no.

\subsection{SOLID}
\subsubsection{SRP — Single Responsibility Principle}
Este principio dice que una clase debe tener una sola responsabilidad principal. En el proyecto se aplicó separando las clases por función.Por ejemplo:

AuthController se encarga del login y registro.

BookPostController se encarga de los libros.

BookCommentController se encarga de los comentarios de libros.

BookReactionController se encarga de las reacciones de libros.

En Flutter también se aplicó:

AuthService maneja autenticación.

BookService maneja libros, comentarios y reacciones.

\subsubsection{OCP — Open/Closed Principle}
Este principio dice que el sistema debe estar abierto para extenderse, pero cerrado para modificarse innecesariamente. En el proyecto se aplicó cuando se agregaron comentarios y reacciones a los libros. En lugar de meter toda esa lógica dentro de BookPostController, se crearon clases separadas:

BookComment, BookReaction, BookCommentController,
BookReactionController, BookCommentRepository,
BookReactionRepository

Esto permitió extender la funcionalidad del sistema sin sobrecargar la clase principal de libros.
